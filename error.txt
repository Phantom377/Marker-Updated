Benefits of Header-Based Chunking
1. Respects Document Structure
The document's own authors decided where section boundaries lie. Using those same boundaries as chunk boundaries means each chunk is a semantically coherent unit — not an arbitrary slice of text cut off mid-sentence or mid-topic. The LLM receives a complete, self-contained section every time.
2. Stays Within Token Limits
Large policy documents can run to thousands of words. Sending the full document in one prompt risks exceeding the model's context window, which either causes an error or forces the model to truncate its reading. Chunking ensures each prompt is well within limits, and the model can give its full attention to the section at hand.
3. More Focused, Higher Quality Questions
When a model is given a 5,000-word document and asked to generate questions, it tends to produce generic questions that skim the surface. When given a single 400-word section on co-payment structure, it produces precise, detailed questions specific to that section. Chunking dramatically improves question relevance and specificity.
4. Easier Debugging and Traceability
If the model produces poor questions for one section, it is immediately clear which chunk was the problem. Each chunk's output is labelled with its section title in the JSON output, making it straightforward to identify, investigate, and reprocess individual sections without re-running the entire document.
5. Parallelisation Potential
Because chunks are independent of one another, they can in principle be processed in parallel — sending multiple API calls simultaneously rather than sequentially. This can significantly reduce total processing time for large document libraries.
6. Section-Level Granularity in Output
The chunked approach produces a nested JSON output organised by section. This is directly useful for chatbot testing because it allows the QA team to test the chatbot's knowledge of specific policy sections, not just the document as a whole. A test failure can be traced back to a specific section, making it easier to identify gaps in the chatbot's training data.
7. Handles Long Documents Gracefully
A document with ten ## sections becomes ten focused prompts. Whether the document is 2 pages or 20 pages, the pipeline handles it consistently — the number of API calls scales with the number of sections, and each call remains predictably sized.

Markdown for Standard LLMs vs. PDF in Microsoft Copilot

Overview
When building AI pipelines that process documents, the choice of input format is not just a technical detail — it directly affects the quality, cost, and reliability of the output. Two common approaches are feeding converted Markdown text to a standard LLM via API, and uploading the original PDF directly to Microsoft Copilot, which has built-in document understanding capabilities. Both work, but they are suited to different contexts and come with different trade-offs.

Markdown with a Standard LLM
How It Works
In this approach, the PDF is first converted to Markdown using a tool such as pymupdf4llm or marker. The resulting Markdown text is then included directly in the prompt sent to the LLM — either as the full document or as chunks split by header. The model reads the text as part of its context window and generates a response based on what it has been given.
The model has no special knowledge of the original document. It only knows what is in the prompt. If the Markdown conversion was accurate, the model works with a clean, structured representation of the policy. If the conversion introduced errors — garbled tables, merged paragraphs, missing headers — those errors are passed directly to the model and reflected in its output.
Strengths
Full control over the pipeline. Every step is explicit and inspectable. You can read the Markdown, verify it looks correct, adjust the chunking logic, and tune the prompt before any API call is made. There are no black boxes.
Consistent, reproducible behaviour. Because the text is injected directly into the prompt, the same input always produces a deterministic response (at a given temperature setting). This is important for testing pipelines where consistency matters.
Works with any LLM. The Markdown approach is model-agnostic. The same converted files and prompts can be used with GPT-4o via Azure, Claude via Anthropic's API, Llama via Ollama, or any other model. You are not locked into a single vendor's ecosystem.
Cost transparency. You know exactly how many tokens you are sending because you control the text. Chunking allows you to keep costs predictable by limiting prompt size per API call.
Programmable and automatable. The entire pipeline — reading files, chunking, calling the API, writing JSON output — can be scripted and run automatically on a schedule or triggered by a webhook. No manual steps are required.
Handles bulk processing efficiently. A folder of 50 policy documents can be processed overnight with no human involvement. Each document is read, chunked, queried, and its output saved to a JSON file automatically.
Weaknesses
Conversion quality is a dependency. The pipeline is only as good as the Markdown conversion. Complex PDFs with multi-column layouts, embedded images containing text, scanned pages, or heavily styled tables may not convert cleanly. Errors in the Markdown silently degrade output quality.
No visual understanding. A standard LLM reading Markdown text cannot interpret charts, diagrams, or images embedded in the original PDF. If a policy document contains a benefits table rendered as an image rather than text, that information is invisible to the model.
Context window limits. Very long documents must be chunked or summarised before being sent. While chunking is manageable, it requires additional engineering and means the model never has a view of the entire document at once.
Prompt engineering overhead. Getting reliable, well-structured output requires careful prompt design. This takes iteration and expertise, and prompts may need to be adjusted as documents vary in structure.

PDF Upload in Microsoft Copilot
How It Works
Microsoft Copilot — available in Microsoft 365 apps such as Word, Teams, and the Copilot web interface — allows users to upload a PDF directly. Copilot uses Azure AI Document Intelligence (formerly Form Recogniser) under the hood to parse the PDF before the LLM ever sees it. This parsing step extracts not just text, but layout information including heading levels, table structure, and the spatial relationship between elements on the page. The LLM then receives a structured representation of the document, not raw bytes.
The user interacts conversationally — asking questions, requesting summaries, or requesting comparisons — and Copilot responds based on its understanding of the uploaded document.
Strengths
Native PDF understanding. Copilot's document parsing layer handles the conversion step automatically and uses Microsoft's production-grade OCR and layout analysis. For well-formatted PDFs, this produces high-fidelity extraction including tables, headings, and multi-column text, without any manual conversion work.
Handles scanned documents. Because Copilot uses OCR, it can extract text from scanned PDFs that are essentially images of pages — something that basic PDF-to-text libraries cannot reliably do.
No technical setup required. A non-technical HR team member can upload a PDF and start asking questions immediately. There is no pipeline to build, no code to write, and no server to run. The barrier to entry is extremely low.
Conversational interface. Users can ask follow-up questions, request clarifications, and explore the document interactively. This is well-suited to ad-hoc queries where the user does not know in advance what they want to ask.
Integrated into existing workflows. For organisations already using Microsoft 365, Copilot is accessible directly within Word, Outlook, and Teams. There is no context switching to a separate tool.
Visual element awareness. Copilot's document intelligence layer can recognise and describe charts, images, and diagrams that are part of the PDF layout, giving it an advantage over text-only Markdown pipelines when documents contain visual content.
Weaknesses
No programmatic control. Copilot is an interactive product, not an API you can call in a script. It cannot be triggered automatically, cannot process a folder of 50 documents overnight, and cannot write its output to a structured JSON file. Every interaction requires a human in the loop.
Output is conversational, not structured. Copilot returns natural language responses. If you need structured output — such as a JSON array of 25 questions with consistent field names — you would need to manually copy, reformat, and validate the output. There is no guarantee of consistent structure across sessions.
Vendor lock-in. The approach only works within the Microsoft 365 ecosystem. It cannot be replicated with a different LLM or on a different platform without significant rework.
Limited transparency. You cannot inspect exactly what text the document parser extracted, how it structured the content, or what was passed to the model. If the output is wrong, it is difficult to diagnose whether the fault lies in the PDF parsing, the model's reasoning, or the prompt interpretation.
Cost model is different. Copilot is licensed per user per month as part of Microsoft 365 Copilot, rather than being billed per token. For large-scale automated processing, this model is inefficient — you are paying for a user licence regardless of how much or how little processing is done.
Context window constraints still apply. Very large PDFs may be truncated or summarised by Copilot before being presented to the model, and you have no control over how this is handled. With a Markdown pipeline you decide the chunking strategy; with Copilot, Microsoft decides.
Not suitable for automated testing pipelines. Because Copilot is interactive and conversational, it cannot be integrated into a CI/CD pipeline or automated testing framework. It is a productivity tool, not an engineering component.

Which to Use
The two approaches are not in competition — they serve different purposes and are often used together within the same organisation.
Use Markdown with a standard LLM when you need automation, reproducibility, and structured output at scale. This is the right choice for the HR question generation pipeline, where dozens of policy documents must be processed programmatically and their outputs written to JSON files for chatbot testing.
Use PDF in Copilot when a non-technical team member needs to explore a single document conversationally — asking ad-hoc questions, generating a quick summary, or checking a specific clause without any engineering overhead. It is a productivity accelerator, not a pipeline component.
For the HR chatbot testing initiative specifically, the Markdown pipeline is the appropriate foundation. Copilot can serve as a complementary tool for the HR team's day-to-day document queries, but it cannot replace the automated, structured question generation that the testing framework requires.
